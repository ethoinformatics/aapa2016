    Goals

    A handheld app that is...

    • Built on open source and standard web technologies
    • Configurable, modular, and allows multiple data streams
    • Runs on multiple platforms and older devices
    • Automatic geolocation tagging and time stamping
    • Works offline without a continuous web connection

    • Flexible enough to support different existing schemas
    • Standards compliant and linked data friendly

    Goals

    A browser-based admin tool that is...

    • Built on open source and standard web technologies
    • Allows and integrates multiple users per project

    • Makes sharing data easy

    Caveat

    We're not (yet!) concerned with designing a slick interface for behavioral data entry...

    ...instead, we're focusing on designing a modular, customizable framework for collecting multiple data streams using a single app and device

    Design Philosophy

    Custom data structures

    • Information that goes together (metadata + data) is flexibly packaged into user-defined data structures called documents
    • Documents are made up from a set of key:value pairs, where keys either are EthoCore terms or have have well-defined relationships with those terms
    • Documents are project-specific "data structures" built by packaging together information from one or more class
    • Not all terms contained in the classes that make up a document need be stored in any given document (i.e., the equivalent of empty fields in a record can be omitted)
    • Documents are (roughly) equivalent to the rows or records in a tabular/relational database except that each is a separate entity
    • The are very similar, too, to the nodes in a graph database.

    Design Philosophy

    Structure of documents

    • Contain a URI
    • Contain key metadata
      • Who made it
      • Datetime (point or series) to which the data pertain
      • Geolocation (point or series) to which the data pertain
    • Represented in a structured text format (JSON)
    • Many of the terms used to make up a document are metadata terms
    • Every document is a resource with a URI
    • Documents are represented in JSON format in a document oriented database.
    • We think many of these are likely to be standard across projects (e.g., “contacts”, “focal samples”, “biological samples”, “feeding trees”), but projects need not use them all and may extend the list of document types

    Example: "contact" document

    Mapping to EthoCore

    Design Philosophy

    Documents can be linked to one or more other documents

    • These links are also resources, also have URIs, and are also stored as documents
    • Some relationships are required (e.g., “focal observations” must be linked to a “focal sample”)
    • Many relationships among documents are flexible and can differ among or within projects
    • These are somewhat like JOIN tables in a SQL-based database, but note that PRIMARY KEY-FOREIGN KEY relationships among tables are not specified
    • These are analogous to EDGES in a graph database and to the predicates in an RDF data store
    • An EthoCore compliant database can be thus schema-less and can accommodate multiple schema

    Implementation

    The framework of our mobile is app is built using...

    ...which are the primary tools used to build interactive web interfaces.

    The real programming work is done in JavaScript...

    ... and we use the node.js framework and the node package manager npm to bundle code from various programming libraries.

    We use the Cordova framework to build the user interface...

    ... which lets us write one application that can be used on iOS devices, Android devices, or other laptops or tablets through a browser.

    And we use document-based NoSQL databases for data storage, both locally, in the app, and on a remote server.

          
    CouchDB is our main database, because it's powerful, yet easy to learn. It also has a mobile version, pouchDB, which integrates well with our mobile framework, Cordova. Couch and Pouch are designed to manage data sync between devices with intermittent network access, so it works well for our purposes.

    Closing thoughts

    1. The framework we are building is very generalized and allows for flexibility and customization in data structures and UI

    2. Use of the EthoCore vocabulary facilitates comparability and data sharing across studies and provides promise for linking to data across the web

    3. The document-oriented, structured text format facilitates data interchange using standard protocols and, simultaneously, it is human-readable and easily recast into other formats

    4. The timeline and map view tabs allow users to access to real time visual feedback in the field

    5. We still long way to go before the app is ready to be deployed!

    General Idea

    • App generates documents that are made up of EthoCore terms
    • Each document is a resource with a URI (uniform resource identifiers)
    • Documents can be decomposed to triple-stores and used as linked data
    • Documents can link to one or more other documents via standard predicates, each of which has a URI (taken within, belongs to, child of, associates with)
    • Media (pictures, audio recordings) can be linked by a predicate to one or more documents
    • Every document has key metadata
      • Who made it
      • When it was created
      • Time (point or series)
      • Geolocation (point or series)
    • History of revisions of documents is preserved

    Some Design Decisions

    • Use HMTL, JavaScript, and CSS to create a mobile app that can be compiled and deployed on a variety of devices
    • Use JSON documents to configure aspects of the application
    • Use local storage and PouchDB on the mobile device to hold data
    • Data in PouchDB stored as JSON documents
    • Sync from PouchDB to CouchDB on a local server and from CouchDB on local server to remote server
    • Use data from operating system's location services to tag all documents with geolocation
    • (from internal GPS, cell towers, external GPS)
    • Geolocation data is updated recorded continuously

    What’s good about documents...?

    • They are basically just text files (though they can include pointers to associated files, like photos or recordings)
    • Standard protocols exist for transferring them quickly and efficiently over the web (e.g., HTTP)
    • Programmatically accessible (APIs) by a wide range of useful technologies (such as in the web)
    • Standard formats exist for “document” data that are at once human *and* machine readable (e.g., JSON, XML)

    The Etho Grammar Data Model

    1. Data consist of documents (a.k.a. resources)

    • Documents group together a set of terms that are drawn from 1 or more classes
    • Documents correspond (roughly) to the rows or records in a tabular database, except that each is a separate resource
    • Many of the terms used are metadata terms
    • Not all terms comprising the classes that make up a document need be stored in any given document (i.e., the equivalent of empty fields in a record can be omitted)

    The Etho Grammar Data Model

    2. There are different types of documents

    • We think many of these are likely to be standard across projects (e.g., “contacts”, “focal samples”, “biological samples”, “roll call census”) but projects need not use them all and may extend the list of document types

    The Etho Grammar Data Model

    3. Documents can be related to other documents

    • Some of those relationships are required (e.g., “focal observations” must be linked to a “focal sample”)
    • Many relationships among documents are flexible and can differ among or within projects

    The Etho Grammar Data Model

    4. Relationships between documents constitute separate resources, also stored as documents

    • These are somewhat like JOIN tables in a SQL-based database, but note that PRIMARY KEY-FOREIGN KEY relationships among tables are not specified
    • These are analogous to EDGES in a graph database
    • An Etho Core compliant database can be thus schema-less and can accommodate multiple schema
